Introduction

So you want to port Flash.

 

We at Middlesoft have attempted to make this job a straightforward as possible. We have separated the Flash code into platform specific (Native) and platform independent parts so a third party can concentrate on very specific and well defined tasks.

 

This document is intended as a guidebook to porting Flash. It is one of 3 documents. This document is the Roadmap to the code, and is intended to be a friendly step by step guide. You will also find a Reference (in html) which is very useful for defining all the method calls you need to port, and a Design document, which gives a “big picture” technical view of the code.

 

This document is a suggested series of steps for getting through the experience of porting Flash. If you are going to suffer through porting someone else’s software (even software as good as Macromedia’s) you may as well have a friendly guidebook. Most of this document is based on the Lead Engineer’s notes as he ported Flash back to Windows – so hopefully it is an accurate guide and warning to the caveats you may encounter.

 

This is also a good place to point out that you do not have the Macromedia Flash player. It is a Flash player, and the core code is almost unchanged from Macromedia’s, but the platform dependant code was mostly written here at Middlesoft. The point is, the windows player now in your hands is not quite as fully tested as the shipping one. But it is much easier to port and understand.

Demo Limitations

Should not affect the core code or Flash player in any way. But these are limitations of the Win32 Demo code.

·        There is nothing wrong with either core or interface support for 1, 2, and 4 bit images. If you port to such a platform, there should be no problem. The demo code probably will not work for these bit depths, however.

·        The file streaming code is only implemented for files on a local or network drive. The implementation of an internet communication layer is 1) completely platform dependent code, and 2) only clutters the example code.

Getting Started

In a moment, you will need to write a main() and insert some basic OS code to get started. But it may be easier to organize your files first, in order to see where everything goes. The main Flash directory contains subdirectories for libraries and versions. The main directory you will (hopefully) not have to change. The subdirectories are platform dependant, however, and will need to be modified. The libraries are:

·        JPeg, which contains the Independent JPEG Group image compression code. The decompression files are used by Flash. Their web page www.ijg.org contains the original source, which in turn contains instructions for compiling on various platforms. It may compile cleanly on your system, but look at jconfig.doc for configuring the jpeg code. The master document, libjpeg.doc, is also provided. Note that if you install new source from the IJG, you only need the decompression routines.

·        ZLib which provides lossless image compression. The ZLib page is at http://www.cdrom.com/pub/infozip/zlib/. ZLib is so “C” that is should compile without incident on anything...but plenty of documentation is on the web page, as well as source code for bunches of other versions. Again, Flash only uses the decompression.

 

It is very likely these libraries will recompile on your system. In the case they do not, both products have substantial documentation to help out. (You may want to delay this task until you have initially ported the Flash platform specific code.)

 

The remaining subdirectory is labeled Win32 and contains the platform specific code for a Windows95/98/NT version of Flash. Create a new subdirectory (off of main) with the name of your platform, and copy the Win32 code into it. You don’t need the Win32 code of course, but we find it’s nice to have a template to work from. And, admit it, it feels good to go through and delete lines of code.

 

The paths to the native code need to be set. The file global.h contains the paths to the native code. It’s a quick job to change these to the paths you are using.

 

Now that the directory structure is correct, it is time to write a basic window and event handler. In the windows case, this was done with the Visual C++ app wizard. Using GTK in Linux, you probably want to make a basic window with menus and a client area. However you do it, In this code you need to new a NativePlayerWnd and delete it at window shutdown. In the Win32 example, the NativePlayerWnd is new’d during the CREATE window message, and delete’d on the DESTROY message.

 

NOTE: The file stdafx.h is hanging out in the main directory, inappropriately, since it is used by the jpeg code as well as the Win32 code.

The ZLib and JPeg code are provided as source. The object library Flash needs is Fraunhofer’s mp3 library. Although if you do not have SOUND and FLASHMP3 defined, you will not need it.

 

The code as delivered has all its #defines turned on. To get started, you want to turn most of them off (except DEBUG and your machine id: WIN32, LINUX, etc.). The following is a list of #defines Flash uses and what they mean. After the name of the #define is either the filename where it is located or the word “compiler” which means it should be set as a compiler option. (The –D option, often.). The full list is in the Design document.

 

DEBUG or _DEBUG (compiler)

Used if in debug mode, which you should be at this point

WIN32,_WINDOWS (compiler)

Or whatever is appropriate for your environment. Affects the JPeg and ZLib.

FIXEDUSEFLOAT (fixed.h)

To get you running, commenting in this #define will turn off fixed point math.

USE_JPEG (compiler)

Remove to turn off the JPEG support. (You don’t need the Jpeg library when this is commented out.)

EDITTEXT (compiler)

Remove to turn off editable text.

SOUND (compiler)

Remove to turn off sound. You very much want to remove this.

FLASHMP3 (compiler)

Even once sound is working, removing this will remove MP3 support. Without the Fraunhofer library, you can’t #define this or use MP3.

 

Porting the Code, Step by Step

Hopefully, you now have a makefile or something equivalent. Now, file by file, we’ll go through the code. You’re a ways from being able to fully build again, but the next goal is to get something compiling.

Debug

We suggest starting the port at the easiest possible point: debug.cpp is a good choice. (There is usually enough issues with the project file and understanding the code without needed to complicate it with a more difficult example.) Debug.cpp defines a couple of straightforward debugging functions. You’ll need to write code, but it is pretty standard code. (The quick solution is to turn off debugging. This would be A VERY BAD IDEA. If you don’t have a debugger to help try at least to have FLASHASSERT core dump with a helpful message.)

 

Once the assert and print functions are written, you should be able to compile the file and feel good about having this process rolling.

Util

Util.h/.cpp (currently) has one function – get the current time. The windows version doesn’t have any code in the cpp (it’s all in the header.) This was not the case for other versions, so the cpp file is left in the project.

Graphics

Display

There are two classes in display.h/.cpp, NativeDisplayTester and NativeBitmap.

 

NativeDisplayTester tests the display mode and maintains information about the display device. For systems with one display mode this is a fast port. Even for more general systems, a good place to start is to return values for your development system. Given the choice, the least work to seeing Flash run is to implement the true color mode.

 

The other end of the spectrum is a smart tester which can handle changing palettes, resolutions, and bit depths, all why selecting the fastest operating mode. When Flash core detects the palette (or graphics mode) has changed, it destroys the current bitmap and constructs a new one. The decision about which kind to use is drawn from NativeDisplayTester.

 

NativeBitmap is a bitmap in memory…but more. It needs to be a bitmap in the sense that Flash will write directly to the bits in memory. (And given a choice, they should be in fast memory as well.) The reason that this is a native class is that its most important function is BltNativeToScreen, which may require additional OS dependent data structures beyond what is required to define a simple bitmap in memory.

 

When figuring out how to implement this class, perhaps it is easiest to start with the BltNativeToScreen call. Once you have figured out what you need to do to get that to work, it will hopefully be clear what kind of bitmap object you need to create.

 

There are the 4 steps to create a bitmap in memory with the same parameters as the client window. All of these steps are performed in core code:

1.      The current screen bit depth is requested from NativeDisplayTester.

2.      If the screen depth has changed, or this is the first NativeBitmap, the screen depth is tested by NativeDisplayTester for the best mode.

3.      If the screen depth is <= 8 bits/pixel, the current color palette is retrieved from NativePalette.

4.      A new NativeBitmap is created from this info.

 

So if the result of this approach is that a resize, palette change, or screen mode change can all be handled by deleting the bitmap. (Accomplished by the NativePlayerWnd::FreeBuffer call.) The bitmap will be constructed again when Flash draws. Kudos to Macromedia for a very nice system.

 

Another important result of this approach is that if you have a fixed size client window, you simply need to not respond to AdjustWindow method calls. AdjustWindow is a request to resize the client window, but if the resize does not occur, Flash will work fine.

 

So after you have defined the bitmap and tester, it gets compiled, and a string of error messages come from the header. Do not panic. Delete the offending Windows code (mostly structures and private data) that is in the Native headers. (Since all the Win32 code is platform dependent, you’ll have to rewrite it anyway.)

Palette

So your mind is wrapped around the graphics problem. Onward to palettes. (If your system doesn’t support palettes, onward to the next section.)

 

There is only one method: GetScreenPalette. Writing the method, hopefully, is straightforward. For the sequence of events that this call is a part of, look at the 4 steps in the previous section.

 

If:

·        Your system has a fixed palette. Then GetScreenPalette should return this palette.

·        You system automatically handles an arbitrary palette. GetScreenPalette should return an “ideal palette”, which is a color cube that has as wide range of colors as possible in 256 entries.

·        You manage the palette. GetScreenPalette should return the current screen palette, but you should attempt to set an ideal palette for your application. (This is what the windows code does.) Palette changes should result in a call to NativePlayerWnd::FreeBuffer.

 

The SetStandardColors function is an example of creating an ideal palette.

 

Once you get this working, take a well deserved break.

Main Widget

NativePlayerWindow

Is a big class. There are 3 parts to this class:

1.      General Methods. Functions that the Flash core code makes to the NativePlayerWindow. Requests to start and stop timers, get the size of the Client (drawing) window, set the Client (drawing) window size, and set the menu status.

2.      Event Handling. As the user and OS generate events (mouse, keyboard, repaint) these events need to be hooked up to Flash.

3.      Streaming/GetURL. The network and file loading layer.

 

The first decision is to decide an architecture. There are 2 obvious choices:

1.      Put all this code in NativePlayerWnd. In the example provided, WinMain and an event handler (WndProc) are functions registered with the OS. WndProc new’s and delete’s a NativePlayerWnd, and uses method calls to communicate with it. This is the obvious approach, and it works well in the example code. There are 2 sub-choices as well:

·        Make the event loop/OS callback a member function of NativePlayerWnd.

·        Make the event loop/OS callback a separate function altogether.

2.      Make a child class of NativePlayerWnd to deal as an OS interface layer, and limit the methods in NativePlayerWnd to those required. This can make NativePlayerWnd a neater interface and easier to deal with, or you can spend so much time hooking up method calls between objects that it’s not worth it.

 

The first time we ported Flash it was to an application framework, and we started with method 1. It quickly became unwieldy, and we switched to method 2. Method 2 was such a success, that we used the same structure when we ported back to Win32, where it was a clunky mess and we switched to method 1.

 

The point being, we don’t have good guidelines on what to choose, but give it some thought. In our experience, it was not hard to switch from one configuration to the other. Also keep in mind there will be special needs if you want, say, multiple player windows in the same application. So, for example, if you wanted multiple flash player running from one event loop, you would probably want more code in NativePlayerWnd (method 1), where multiple copies are new’d from the event loop. (Which we haven’t tried. But that is where we would start.)

 

Note that there are a couple of instances of backcasting from SPlayer to NativePlayerWnd. So you don’t want to break that hierarchy unless you really need to.

 

Things you must port to get it to work:

 

StartTimer and StopTimer

There are 2 timers started (or stopped) by the start (or stop) call. Implementation of the timer is up to you. They are repeating timers. The Play Timers signals frame updates, and the Cursor Timers signals the blinking of the edit text cursor.

ClientRect

Returns a 0 based rectangle containing the client window size.

File open command

To actually commence the loading of the Flash file.  If you don’t want to implement menus yet, a call to ControlOpen( char* url ) after the app is running will load and play a SWF file.

InvalidateScreenArea

Requests an OS repaint message be sent for the (0 based) SRECT passed in. It is acceptable to generate a message for the  entire client window. The repaint is expected to occur after execution has returned to the NativePlayerWnd, not while processing the InvalidateScreenArea call.

AdjustWindow

Sets the size of the client window. This should do nothing for a fixed size window.

Event: Window Resize

When a resize message occurs, FreeBuffer needs to be called.

Event: Window Paint

When a paint message arrives, it should call Repaint.

Streaming (NativePlayerWindow continued)

Not a small task. But it is pretty fast to just get “up and running” so we’ll cover that first.

 

If you assume that the URL is a file that can be open’d with the C library function fopen, then the number of possible cases to be dealt with is much smaller. Try opening the URL as a local file; if that succeeds then the following code will send the SWF stream – immediately – to Flash.

StreamData streamData;

char block[256];

int  read;

 

StreamInNew( &streamData, notifyData );

while ( read = fread( block, 1, 256, fp ) )

{

StreamInWrite( &streamData, &block, read );

}

StreamInDestroy( &streamData );

 

This code was pulled from PlayerWnd.cpp, where you can see it in its entirety.

 

At this point, you should be able to build all and run Flash! Congratulations!

Now That Flash Is Working…

Streaming, Part II

A more complete implementation of streaming involves writing the code to connect to the Netscape plug in interface. Needless to say, this is substantially more challenging if you are not a plug-in and have to write the network connection code yourself. Arm yourself with the Netscape docs, the Design doc, and knowledge of threads on your system. The demo code provided uses a second thread to stream local files; this demonstrates streaming on a separate thread. But there is still work to be done to implement the entire Get and Post interface – if you need that for your system.

NativePlayerWindow, Part II

The tasks to finish up the NativePlayerWnd should be completed in the following order:

Mouse and Keyboard handling

Event handling (mouse and keyboard) is the somewhat tedious but generally straightforward process of connecting operating system events to method calls in the flash core code. You’ll need to define EDITTEXT to use the keyboard in a meaningful way.

Cut and Paste

Now that the Keyboard is working, the mouse is working, and EDITTEXT is turned on, the final step is cut and paste support. There are only two calls to be implemented: GetStringFromClipboard and SetStringToClipboard. Flash can support either ASCII or Unicode, depending on whether EDITTEXT_CLIPBOARD_WIDE is #defined. Make sure that lines are from the clipboard are \n separated, and strings are null terminated.

 

If you just want Cut and Paste to work in Flash – not with other apps – then simply having a “Clipboard” character string that is set by SetStringToClipboard and retrieved by GetStringFromClipboard works nicely.

UpdateCursor

When Flash feels it is necessary to change the cursor shape, it calls this function. The native code may respond by calling GetCursorType which returns an id of the current cursor. (GetCursorType can be called at any time the native code needs to get the current cursor.) Needless to say, actually setting the cursor image is left completely up to you.

 

Icons and cursors that Flash uses are in the FlashIcon.tga/pcx file.

Menus

There are two menus in Flash, the application menu at the top of the screen, and popup menus that occur in response to a right mouse click. (Or whatever mouse click your system uses.)

 

EnableMenus is called when the core code determines that the status of the application menu items has changed.  If your player does not use menus, you can ignore this call.  For players that use menus, the enabled, disabled, and checked status of the menu items should be set in response to this call. 

 

So when you implement EnableMenus, it does nothing more than change the states of the menu items. So if menuStatus.play in set true, then the Play menu option should be enabled for the user to click.

 

If you need to request the state of the menus, use GetMenuState.

 

The other kind of menu, Popup menus, are more involved than they first appear. When the user right-clicks the following steps should occur in your code:

1)     Get the type of the menu that Flash wants to Popup with GetPopupMenuState.

2)     It will either be MENU_EDIT or something else:

a)      If MENU_EDIT, then:

i)       Find out the enabled/disabled state of the menu items with GetEditMenuState

ii)     Set the enabled/disabled states in the popup menu.

iii)   Wait for the menu selection message to come in, and call HandleKeyDown with the appropriate key. (The Edit menu simulates key presses.)

b)      if not an edit menu, then:

i)       There are 4 flavors of the popup menu. Yowza. They are listed in the Design Document. So construct the correct menu choices based on what type of popup (MENU_NONE, MENU_NO_MOVIE, MENU_NORMAL, MENU_STATIC) you have. There are 16 menu items, potentially, but they are never all on the menu at the same time.

ii)      If you did not cache the information passed in to the EnableMenus call (and there is really no reason to), call GetMenuState to get the enabled/disabled/checked state of the menu items. The popup menu and the application menu have the same state. So of the 16 potential items on the popup menu, any of the items displayed can be enabled or disabled.

iii)   Set the enabled, disabled, and checked states for the popup menu.

iv)    Put that menu on the screen. The user (devious as they are) may select ZoomInContext, in which case you will need the position of the right mouse click in order to zoom in. So save the position of the right click now, just in case.

v)      Wait for the menu selection message to arrive, and call the appropriate Control* function.

Fixed Math

Flash uses fixed math. Above, we defined FixedUseFloat. You may wish to turn this off if your system can perform fixed point math faster than floating point. If your system supports 64 bit integers, there are 2 simple defines to set up, and fixed point math should be ready to go.

 

Examples of assembly code on various processors is in the Fixed.h file.

Sound

Sound is somewhat beyond the scope of this document. Sound does not port well. There are portable sound libraries out there (SDL at http://www.devolution.com/~slouken/SDL/ is a good one) but they have to jump through some hoops to work across platforms. The design document contains a description of how the core and native sound code works.

 

Flash has the concept of there being sound buffers. So the native sound engine requests buffers to be filled, sends the buffers to the hardware, and waits for the buffers to complete playing. As buffer completion occurs, new buffers are requested from the core code.

 

Differences in sound implementations and multi-threading somewhat cloud the issue, unfortunately. As you write the native sound mixer, it will probably be prudent to change the interface to the NativeSoundMix to suit your platform.

Speed Testing

To testing Flash, you can create a SWF file with consistent, looping animation, and set the frame rate at something high. (Say 30fps.) By turning on the FRAMESPERSEC define, the frame rate will print every 12 frames, so you can see the effect of changes and window resizing. Note that the code uses FLASHOUTPUT to print so it will not work in a release build. ScriptThread::DoFrame contains the code if you wish to implement output for a release build.

 

The file speedy.swf is provided with the Porting SDK. Speedy plays back at 30fps, and should be a good stress test.

OS Level Font Support

If you need a smaller Flash player and want a little more performance out of editable text, you may want to use your operating system to draw the edit text.  Basically, you are facing a rewrite of EditText::Draw, a by no means trivial job. But it can be done, resulting in a much smaller and possibly somewhat faster Flash player.

 

Once OS fonts are implemented, the unixfonts.cpp file and the data it contains can be removed from the project.

Version Identification

Macromedia has set up a system in which each version of the player will have a unique version string containing platform and version information.  When you port flash to another system or device, you should modify the version number and the platform string in “native/version.h”.

 

In “native/version.h”, set the major version to reflect the number of the major release of your code and the minor release to reflect lesser revisions of your source code.  The platform string should reflect a unique string representing your platform.  Since this platform string must be unique, you must license it with Macromedia before distribution of your product.  Send an email to Flash_Source@Macromedia.com to obtain approval to reserve your platform id string.

 

This version string is available within a movie with the variable $version, and will take the form, eg., $version = “ABC 4.26”.  Since the Flash authoring tool will not allow you to use the dollar sign in variable names, you can set a variable to the version string as follows:

variable = Eval(“$” & “version”)
